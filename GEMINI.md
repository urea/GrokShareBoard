# Grok Share Board Project

## 概要
Grok Imagineの生成URLを共有・検索・閲覧できる掲示板サイト（キュレーションプラットフォーム）。
X（Twitter）のような動画プレビュー再生、プロンプト検索、ユーザーごとの生成物一覧などを提供する。

## ディレクトリ構成
- `_App/`: Next.js アプリケーションのソースコード
- `_Data/`: データベース (SQLite), ユーザーデータ
- `_Dev/`: 開発用ドキュメント, ツール, メモ
  - `docs/`: 仕様書など

## 技術スタック (予定)
- Framework: Next.js (App Router)
- Styling: Tailwind CSS
- Database: SQLite (better-sqlite3)
- Icons: Lucide React

## 開発ルール
- `_Dev/Tools` の既存ツールを活用する。
- ドキュメントは `_Dev/docs/misc` に `YYYYMMDD_Topic` 形式で保存。
- 日本語で開発・ドキュメント作成を行う。
- **思考プロセスの日本語化**: AIアシスタントが表示するタスク名 (TaskName)、状況 (TaskStatus)、サマリー (TaskSummary) もすべて日本語で行うこと。英語での表示は不信感の原因となるため厳禁とする。
- **「開発」と「リリース」のフェーズ分離と確認義務**: AIは、コード修正が完了した際、「完成したから」と自己判断し**即座に `git commit` や `git push` を自発的に実行してはならない**。コード変更後は一旦作業を止め、ユーザーに対し「ローカル環境で動作確認（テスト）を行うか」あるいは「このままコミットしてよいか」を必ず尋ねること。ユーザーの意図（テストフェーズかリリースフェーズか）を勝手に推測して先走ることは厳禁とする。

## 信頼性およびデータ管理ルール (最重要)
- **質問への即応禁止**: ユーザーからの「〜しましたか？」という問いは、まず現状を正確に確認し、その結果を報告するフェーズを必須とする。即座に「修正」してはならない。
- **データ操作の事前承認**: データの消失・不整合の修復、および大規模な移行作業を行う際は、必ず「何が起きたか」「これから何をするか（プログラムの内容）」を説明し、**ユーザーの明示的な承認を得てから**実行すること。
- **独断の禁止**: 「良かれと思って」の事後報告は信頼を損なう行為であると認識し、作業の節目で必ず対話を行う。

## バージョニングルール
- バージョンの数字をあげずにコミットすることは許されない
- `x.y.z` 形式
- `x` (Major): ユーザーの指示があった場合のみ上げる。
- `y` (Minor): 機能追加時に上げる。
- `z` (Patch): バグ修正時・軽微な修正時に上げる。
- **CHANGELOGの更新とタイミング**: バージョン（x, y, zのいずれか）を上げる修正を行った場合、**必ずコミットを実行する「前」**に `CHANGELOG.md` に変更内容を明記すること。コード修正とCHANGELOG更新を同一のバージョン枠で同期してコミットする。
